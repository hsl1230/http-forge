# Refactoring Validation Checklist

Use this checklist to verify code quality AFTER refactoring.
See: `.refactor-validation.md` in project root.

---

## Pre-Refactoring
- [ ] Branch created: `refactor/[feature-name]-ddd`
- [ ] Current tests pass: `npm test`
- [ ] No uncommitted changes
- [ ] Baseline captured in `.refactor-baseline/`

---

## Architecture Compliance ⚙️

### Dependency Injection
- [ ] No `getServiceContainer()` calls anywhere
- [ ] All class dependencies in constructor
- [ ] All constructor parameters are INTERFACES (not concrete classes)
- [ ] No `new ServiceClass()` except in Factory or where specifically needed

### Type Safety
- [ ] No `as any` type casts
- [ ] All types explicitly defined
- [ ] If casting needed, use specific types or create interfaces
- [ ] ESLint passes: `npm run lint`

### Structure & Separation of Concerns
- [ ] Handler file in: `src/presentation/webview/message-handlers/`
- [ ] Command file in: `src/application/commands/`
- [ ] DTO file in: `src/application/dto/`
- [ ] Orchestrator file in: `src/orchestration/services/`
- [ ] Test file in: `__tests__/` subfolder

### No Circular Dependencies
- [ ] Handler doesn't import from DataProvider that imports from Handler
- [ ] Application layer doesn't import from Presentation layer
- [ ] Orchestration doesn't import from Infrastructure above it
- [ ] Run: `npm run check-circular` (if available) or check manually

---

## SOLID Principles ✅

### S - Single Responsibility
- [ ] Class has ONE reason to change
- [ ] Handler = message routing only (not business logic)
- [ ] Command = use case orchestration (not UI logic)
- [ ] Each class can be described in one sentence

### O - Open/Closed
- [ ] New features can be added without modifying existing code
- [ ] HandlerFactory can register new handlers without changing Panel
- [ ] Existing commands unchanged when adding new ones

### L - Liskov Substitution
- [ ] All implementations of interface can be swapped
- [ ] Handler V2 can replace V1 without breaking code
- [ ] Test with different implementations

### I - Interface Segregation
- [ ] Classes don't depend on methods they don't use
- [ ] Interfaces are small and focused
- [ ] No "god interfaces" with 20+ methods

### D - Dependency Inversion
- [ ] Depends on abstractions (interfaces)
- [ ] Never direct dependency on concrete class
- [ ] Injected dependencies are INTERFACES

---

## Code Quality 📝

### Naming & Structure
- [ ] Files named consistently: `*.command.ts`, `*-handler.ts`, `*.dto.ts`
- [ ] Classes have descriptive names (RequestExecutionCommand, CookieHandler)
- [ ] Methods have descriptive names (execute, handle, validate)

### Documentation
- [ ] JSDoc on every public class
- [ ] JSDoc on every public method
- [ ] JSDoc explains responsibility clearly
- [ ] JSDoc includes @param and @returns for methods

### Error Handling
- [ ] try-catch block in Handler.handle()
- [ ] logger.error() called on catch
- [ ] Error messages sent to Webview
- [ ] Errors not silently swallowed

### Logging
- [ ] logger.debug() at method entry points
- [ ] logger.debug() with context data
- [ ] logger.error() on exceptions
- [ ] No console.log() (use logger instead)

### Immutability
- [ ] DTO Output class fields are readonly
- [ ] DTO Input interface properly documented
- [ ] No modification of passed-in objects
- [ ] No direct mutation of state

---

## Testing 🧪

### Test Coverage
- [ ] All new code has tests
- [ ] Success path tested
- [ ] Error path tested
- [ ] Validation tested
- [ ] 80%+ line coverage
- [ ] Run: `npm test -- --coverage`

### Mock Quality
- [ ] All external dependencies mocked
- [ ] Mocks have correct return types
- [ ] Mocks verify correct calls (toHaveBeenCalledWith)
- [ ] No calls to real services

### Test Organization
- [ ] Tests in `__tests__/` folder
- [ ] Test file name matches implementation (`handler.ts` → `handler.spec.ts`)
- [ ] Tests use Jest framework
- [ ] Each test describes one scenario

---

## Integration 🔗

### Handler Registration
- [ ] New handler added to HandlerFactory
- [ ] Factory passes correct dependencies
- [ ] getSupportedCommands() unique (no conflicts)

### Command Registration
- [ ] Command accessible from Handler
- [ ] Command dependencies properly wired
- [ ] Event bus configured if needed

### Webview Communication
- [ ] Handler responds to correct message type
- [ ] Response message format matches expectations
- [ ] Error response format consistent
- [ ] Test end-to-end flow in VS Code

## File Validation ✔️

### TypeScript Compilation
- [ ] No TypeScript errors: `npx tsc --noEmit`
- [ ] All types properly defined
- [ ] No implicit any

### Linting
- [ ] ESLint passes: `npm run lint`
- [ ] No warnings ignored
- [ ] Code style consistent with project

### Tests Pass
- [ ] All tests green: `npm test`
- [ ] No skipped tests (.skip)
- [ ] No pending tests (.todo)

### Build Success
- [ ] Extension builds: `npm run build`
- [ ] Bundle size reasonable
- [ ] No warnings in build output

## Validation Scripts

Run these before merging:

```bash
# Pre-refactoring (before asking Copilot)
./scripts/pre-refactor-check.sh <feature-name>

# Post-refactoring (after Copilot generates)
./scripts/post-refactor-check.sh

# Manual checks
npm run lint:architecture      # Check for violations
npm run type-check             # TypeScript validation
npm test -- --coverage         # Tests + coverage
npm run build                  # Final build test
```

---

## Red Flags 🚨

If you see ANY of these, STOP and fix:

- [ ] ❌ `getServiceContainer()` anywhere in new code
- [ ] ❌ `as any` in application or orchestration layers
- [ ] ❌ Tests failing after refactoring
- [ ] ❌ Type errors after refactoring
- [ ] ❌ Handler in wrong directory
- [ ] ❌ Missing JSDoc
- [ ] ❌ No tests for new code
- [ ] ❌ Circular dependencies detected
- [ ] ❌ Missing error handling
